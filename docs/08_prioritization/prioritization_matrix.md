# Prioritization Matrix – Phase 2

## Method Used: MoSCoW

The MoSCoW prioritization method is used to classify features based on their importance
for user value and feasibility within Phase 2.

- **Must**: Critical for Phase 2 to be considered complete
- **Should**: Important but not strictly blocking
- **Could**: Nice-to-have improvements
- **Won’t**: Explicitly excluded from Phase 2

## Feature List

| Feature                                      | Priority | Justification |
|----------------------------------------------|----------|---------------|
| Real-time user-to-user messaging             | Must     | Core social interaction feature |
| Add contacts using the application           | Must     | Required to enable communication |
| Share publications                           | Must     | Drives engagement and community interaction |
| Watch marketplace between users              | Must     | Core business and value proposition |
| Integrated payment system                    | Must     | Required to complete transactions |
| Notifications (messages / sales)             | Should   | Improves user experience |
| Advanced watch search                        | Should   | Useful but not essential |
| Ratings and reviews system                   | Could    | Adds trust but increases complexity |
| Group chat                                   | Could    | Not critical for the MVP |
| Multi-currency support                       | Won’t    | Too complex for Phase 2 |
| Offline mode                                 | Won’t    | Not relevant for social features |
